# 技术方案

## 总体架构

MVP 采用 Android 客户端和 Deno 后端两部分：

- 客户端：纯 Android 原生（Kotlin）。
- 后端：Deno + Hono。
- OCR：Android 端使用 Google ML Kit Text Recognition。
- 大模型：优先接入国内 OpenAI 兼容模型服务，并选择具备联网能力的模型。
- 知识库：后端加载人工整理的 JSON 知识库文件。

## 客户端职责

客户端为纯 Android 原生工程（`apps/android`）。`MainActivity` 仅作为启动器：

- App 主界面（仅“开始”按钮，由代码构建视图）。
- 点击“开始”时一次性申请悬浮窗与屏幕捕获权限，启动读屏会话，并把屏幕捕获授权令牌经 `onActivityResult` 转发给业务大脑。

业务大脑由原生类承载，拆分为三块，组成完整单次识别流水线：

- `AssistantController`：编排权限、会话生命周期、识别流水线与状态机（识别中/展开）。
- `FloatingController`：悬浮球与展开菜单（收起态显示状态，展开态触发“识别”“结束”）、系统级顶部覆盖答案卡片渲染。
- `AnswerClient`：调用后端统一答题接口并解析答案卡片。
- `ScreenCaptureService`：屏幕捕获读屏会话（前台服务保活）、当前屏幕帧获取、会话失效主动回退（系统回收授权时回调提示卡片）。

ML Kit OCR 由 `AssistantController` 在端侧直接调用。权限检查与失败状态由原生层展示。

## 读屏会话

读屏会话由用户显式开始。点击“开始”时一次性申请所需权限，缺少权限时跳转系统授权页并阻塞当前操作。会话有效期间，用户可通过悬浮球多次手动触发单次识别。

会话在以下情况结束：

- 用户主动结束。
- Android 系统回收屏幕捕获授权。
- App 进程中断。
- 屏幕捕获失败。
- 前台服务不可用。

会话失效后不自动恢复，用户需要重新开始。

## 单次识别链路

1. 用户点击悬浮球“识别”。
2. 原生进入识别中状态（禁用识别按钮防并发抓帧）。
3. 原生获取当前屏幕画面。
4. ML Kit 在端侧完成 OCR。
5. 原生将 OCR 原文（rawText）发送至后端统一答题接口，并在 `clientContext.timing` 中带上 `captureMs`、`ocrMs` 等客户端分段耗时。
6. 后端命中知识库则直接返回；否则调用联网模型并由模型结构化题干、选项与答案。
7. 原生在顶部覆盖卡片展示答案、依据、置信度与来源。

## 超时预算

端到端超时按段分配，并保持不等式约束 **OCR 段 + 网络段 ≤ 端到端硬上限**、**后端两次模型调用合计 ≤ 网络段**，避免“客户端已超时、后端仍在跑”的资源浪费：

- 端到端硬上限：20s（客户端从触发识别到放弃）。
- 端侧 OCR：8s。
- 网络 + 后端：约 12s（硬上限减去截屏与 OCR 已耗时间）。
- 后端模型单次调用：10s。
- 模型解析失败重试：仅当首次输出无法解析为 JSON 时重试一次，且两次合计纳入后端总预算，不得外溢到客户端掐断之后。

## 后端职责

后端负责：

- 加载 JSON 知识库文件。
- 清洗和规范化题目文本。
- 使用题干和选项进行知识库匹配。
- 在知识库命中不足时调用具备联网能力的模型。
- 要求模型输出结构化答案结果。
- 返回推荐答案、依据、置信度和来源信息。
- 记录最小可观测性日志。

## 知识库匹配

MVP 使用简单文本匹配和相似度策略即可：

- 完全相同题干和选项视为高置信命中。
- 题干相似且选项高度重合视为较高置信命中。
- 只命中题干但选项差异较大时降低置信度。
- 题干和选项都无法稳定匹配时调用联网模型。

## JSON 知识库文件

知识库文件初始由人工整理，并支持运行时自动回填（详见“模型调用”）。条目结构：

```json
{
  "id": "bili-core-0001",
  "question": "题干文本",
  "options": [
    { "id": "A", "text": "选项 A" },
    { "id": "B", "text": "选项 B" },
    { "id": "C", "text": "选项 C" },
    { "id": "D", "text": "选项 D" }
  ],
  "answerOptionId": "B",
  "answerText": "选项 B",
  "explanation": "简短解释",
  "sources": [
    {
      "title": "来源标题",
      "url": "https://example.com"
    }
  ],
  "tags": ["分区", "主题"]
}
```

## 模型调用

MVP 选择国内 OpenAI 兼容模型服务（智谱 GLM），且优先使用模型自带联网能力。后端通过 Provider 边界调用模型，业务接口不直接暴露供应商字段。模型相关参数（baseUrl、模型名、是否联网、温度、超时）集中在 `config/model.json`，API Key 走环境变量 `GLM_API_KEY`（不入库）。未配置 Key 时回退到桩 Provider。

模型需返回结构化答案结果，并同时整理出 `question` 与 `options`。后端使用强约束 system prompt 加 few-shot 示例引导模型只输出 JSON；解析时容错处理（剥离 markdown 包裹、字段缺省给默认、非法置信度降级为 low）。仅当首次输出无法解析为 JSON 时，以更严格的提示重试一次；慢调用或网络失败不重试。

高可信回填：当模型返回 `confidence === "high"` 且带有结构化题干时，后端在响应返回之后异步将该题写入知识库（内存串行队列防竞态，按题干去重跳过），形成快路径缓存。

## 可观测性

后端落一条全链路结构化日志（`kind: "answer_request"`），将客户端与服务端分段聚合在同一 `requestId` 下。客户端分段（`captureMs`、`ocrMs` 等）经 `clientContext.timing` 上传，服务端分段（`kbMatchMs`、`modelCallMs`、`totalServerMs`）由后端测量并合并到 `timing` 子对象。

后端至少记录：

- 请求 ID。
- 请求时间。
- 完整题目文本。
- 是否命中知识库。
- 是否调用模型。
- 模型调用是否成功。
- 模型重试次数。
- 分段耗时 `timing`（客户端 + 服务端）。
- 总耗时。
- 失败原因。
- 返回置信度。

慢请求告警：当 `totalServerMs > 5s` 或本次请求调用了模型时，额外输出一条 `kind: "slow_request"` 日志，便于直接 grep 定位慢链路。

不记录：

- 截图。
- bilibili 账号。
- Cookie。
- 用户身份。
