# 技术方案

## 总体架构

MVP 采用 Android 客户端和 Deno 后端两部分：

- 客户端：React Native + Android 原生模块。
- 后端：Deno + Hono。
- OCR：Android 端使用 Google ML Kit Text Recognition。
- 大模型：优先接入国内 OpenAI 兼容模型服务，并选择具备联网能力的模型。
- 知识库：后端加载人工整理的 JSON 知识库文件。

## 客户端职责

React Native 负责：

- App 主界面。
- 悬浮控制的业务状态编排。
- 单次识别流程编排。
- 调用后端统一答题接口。
- 渲染顶部提示条和展开提示。
- 展示失败状态。

Android 原生模块负责：

- 悬浮窗能力。
- 屏幕捕获读屏会话。
- 前台服务或后台运行支撑。
- 当前屏幕帧获取。
- ML Kit OCR 调用。
- 系统级顶部覆盖提示能力。
- 权限检查和授权结果回传。

## 读屏会话

读屏会话由用户显式开始。开始时统一权限模块检查所需权限，缺少权限时阻塞当前操作并请求授权。会话有效期间，用户可以多次手动触发单次识别。

会话在以下情况结束：

- 用户主动结束。
- Android 系统回收屏幕捕获授权。
- App 进程中断。
- 屏幕捕获失败。
- 前台服务不可用。

会话失效后不自动恢复，用户需要重新开始。

## 单次识别链路

1. 用户点击悬浮控制。
2. 客户端进入识别中状态。
3. Android 原生模块获取当前屏幕画面。
4. ML Kit 在端侧完成 OCR。
5. 客户端得到原始题目文本，并尽力解析题干和选项。
6. 客户端调用后端统一答题接口。
7. 后端返回答案结果。
8. 客户端展示顶部提示条。

## 后端职责

后端负责：

- 加载 JSON 知识库文件。
- 清洗和规范化题目文本。
- 使用题干和选项进行知识库匹配。
- 在知识库命中不足时调用具备联网能力的模型。
- 要求模型输出结构化答案结果。
- 返回推荐答案、依据、置信度和来源信息。
- 记录最小可观测性日志。

## 知识库匹配

MVP 使用简单文本匹配和相似度策略即可：

- 完全相同题干和选项视为高置信命中。
- 题干相似且选项高度重合视为较高置信命中。
- 只命中题干但选项差异较大时降低置信度。
- 题干和选项都无法稳定匹配时调用联网模型。

## JSON 知识库文件

知识库文件由人工整理，不做自动爬取。建议条目结构：

```json
{
  "id": "bili-core-0001",
  "question": "题干文本",
  "options": [
    { "id": "A", "text": "选项 A" },
    { "id": "B", "text": "选项 B" },
    { "id": "C", "text": "选项 C" },
    { "id": "D", "text": "选项 D" }
  ],
  "answerOptionId": "B",
  "answerText": "选项 B",
  "explanation": "简短解释",
  "sources": [
    {
      "title": "来源标题",
      "url": "https://example.com"
    }
  ],
  "tags": ["分区", "主题"]
}
```

## 模型调用

MVP 选择国内 OpenAI 兼容模型服务，且优先使用模型自带联网能力。后端通过 Provider 边界调用模型，业务接口不直接暴露供应商字段。

模型需要返回结构化答案结果。如果模型无法提供稳定来源，后端仍保留来源字段，但可以标记为来源不足并降低置信度。

## 可观测性

后端至少记录：

- 请求 ID。
- 请求时间。
- 完整题目文本。
- 是否命中知识库。
- 是否调用模型。
- 模型调用是否成功。
- 总耗时。
- 失败原因。
- 返回置信度。

不记录：

- 截图。
- bilibili 账号。
- Cookie。
- 用户身份。

- 接入可控搜索 API。
- 实现搜索来源白名单。
- 增加答案缓存。

- 引入本地高频题缓存。
