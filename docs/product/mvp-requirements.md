# MVP 需求说明

## 目标

构建一个运行在 Android 手机上的 bilibili 硬核会员答题助手。用户在 bilibili 知识测试页面中，通过悬浮控制手动触发单次识别，应用读取当前屏幕文字，向后端请求答案结果，并在屏幕顶部展示答案提示。用户自行完成作答。

## 用户流程

1. 用户启动 App。
2. 用户开启悬浮控制。
3. 用户在需要答题时进入读屏会话；如果缺少系统权限，统一权限模块按需请求授权。
4. 用户在题目页面点击悬浮控制，触发一次单次识别。
5. Android 端截取当前屏幕画面，并在端侧完成 OCR。
6. 手机端尽力解析题干和选项，同时保留原始题目文本。
7. 手机端向后端统一答题接口提交题目文本。
8. 后端先查询后端知识库；命中不足时调用具备联网能力的模型服务。
9. 后端返回结构化答案结果。
10. 手机端通过顶部提示条展示推荐答案、置信度和简短依据。
11. 用户根据答案提示自行选择答案。

## MVP 范围内

- Android 端优先实现。
- React Native 编写主要界面和业务编排。
- Android 原生模块实现悬浮控制、屏幕捕获、后台运行、系统覆盖提示和 OCR 桥接。
- 用户手动触发单次识别。
- 端侧 OCR，只上传识别出的题目文本，不上传截图。
- 只支持单选题。
- 手机端可以尽力拆分题干和选项，但后端必须接受原始题目文本。
- 顶部提示条默认紧凑展示，尽量避免遮挡题干和选项。
- 答案提示包含推荐答案、简短依据和置信度。
- 低置信答案允许展示，但必须明确标注依据不足。
- 后端知识库由 JSON 知识库文件人工整理导入。
- 知识库条目包含题干、选项、答案、解释、来源和标签。
- 正确答案同时保存选项文本和选项字母，匹配以选项文本为主。
- 后端采用统一答题接口。
- 后端记录最小可观测性日志，可以包含完整题目文本，但不记录截图或用户身份。

## MVP 范围外

- iOS 支持。
- 自动答题或自动点击选项。
- 持续监听、自动扫题或自动识别题目切换。
- OCR 文本确认或编辑入口。
- 上传完整屏幕截图。
- 接入 bilibili 账号、Cookie、私有接口或页面 DOM。
- 用户登录和账号系统。
- 用户侧答题历史。
- 管理后台。
- 自动爬取题库。
- 答案缓存。
- 限流和成本控制。
- 多选题、判断题、填空题和问答题。
- 流式答案展示。
- 后端自建搜索 API 和严格搜索来源白名单。

## 状态与失败处理

悬浮状态包含未启动、待识别、识别中、展示结果和失败。

失败提示需要区分原因，文案保持短：

- 缺少悬浮窗权限。
- 缺少屏幕捕获授权。
- 读屏已停止，请重新开始。
- 未识别到题目。
- 网络不可用。
- 后端请求失败。
- 暂未找到可靠答案。

读屏会话因用户结束、系统回收授权、应用进程中断或捕获失败而失效时，不自动恢复。悬浮控制应回到未启动或失败状态，并提示用户重新开始。

## 隐私边界

- 手机端不上传截图。
- 后端不接收 bilibili 账号、Cookie 或登录态。
- 后端日志可以记录完整题目文本，用于质量分析和错误排查。
- 后端日志不记录用户身份和完整屏幕图像。
